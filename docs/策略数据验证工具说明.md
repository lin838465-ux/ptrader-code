# 策略数据验证工具说明

目的：让策略规则建立在真实历史数据上，而不是拍脑袋。

关联工具：
- [strategy_data_validator.py](C:\Users\mengx\Desktop\量化策略\策略源码\tools\strategy_data_validator.py)
- [rules_etf12_example.json](C:\Users\mengx\Desktop\量化策略\策略源码\tools\rules_etf12_example.json)

## 一、这个工具解决什么问题

以后不管你写 ETF、小盘股还是别的策略，都应该先验证两件事：

1. 数据本身对不对
- 是否有重复日期
- 是否有空值
- 是否有异常价格
- 是否有大段数据缺失

2. 规则是否真的有数据支撑
- 比如“当天涨超 5% 就不买”是否真的合理
- “3 日涨很快但 20 日位置不高”是不是反而可以接受
- “20 日累计涨很多再加速”是不是更容易回调

## 二、推荐的数据来源

优先级建议：

1. 券商 / PTrade 导出的历史数据
- 这是最优先来源
- 因为和实盘下单使用的数据口径最一致

2. 研究环境里通过 `get_history` 导出的历史数据
- 也可以
- 关键是要和策略实盘看到的数据口径尽量一致

3. 第三方网页或财经站数据
- 只能辅助参考
- 不建议直接拿来定最终实盘阈值

结论：
- 你说得对，最终规则应该用券商/PTrade 历史数据来验证

## 三、输入数据格式

当前工具支持一个 csv，至少包含这 3 列：

- `date`
- `code`
- `close`

推荐再包含：

- `open`
- `high`
- `low`
- `volume`
- `amount`

示例：

```csv
date,code,open,high,low,close,volume
2025-10-08,159992.SZ,0.812,0.830,0.808,0.828,1234567
2025-10-09,159992.SZ,0.829,0.845,0.822,0.840,1567890
```

如果是多个标的，直接放在同一个 csv 里就行。

## 四、规则配置格式

规则文件是 json，核心字段：

- `name`：规则名字
- `filter`：pandas query 表达式
- `min_samples`：最小样本数

示例：

```json
{
  "rules": [
    {
      "name": "当天涨超5且20日涨超12",
      "filter": "ret1 > 0.05 and ret20 > 0.12",
      "min_samples": 5
    }
  ]
}
```

当前支持在规则里用这些字段：

- `ret1`
- `ret3`
- `ret5`
- `ret10`
- `ret20`
- `ret60`
- `bias5`
- `bias10`
- `bias20`
- `fwd1`
- `fwd3`
- `fwd5`
- `fwd10`

含义：
- `ret20`：过去 20 个交易日涨跌幅
- `bias5`：相对 5 日均线的偏离
- `fwd5`：从当天收盘买入后，未来 5 个交易日收益率

## 五、如何运行

示例命令：

```powershell
python .\tools\strategy_data_validator.py `
  --data .\data\broker_etf_history.csv `
  --rules .\tools\rules_etf12_example.json `
  --outdir .\validator_output\etf12
```

运行后会生成：

- `data_quality.json`
- `rule_results.json`
- `report.md`

## 六、输出怎么看

### 1. 数据质量报告

主要看：
- 有没有重复记录
- 有没有空收盘价
- 有没有非正价格
- 每个标的最大日涨跌幅是否异常
- 数据时间覆盖是否完整

### 2. 规则结果报告

主要看：
- 样本数够不够
- 命中后未来 1/3/5/10 日平均收益
- 命中后未来 1/3/5/10 日下跌比例

如果一个规则：
- 样本少
- 未来收益没明显变差
- 下跌比例也没显著上升

那这个规则就不该直接用于实盘。

## 七、以后开发策略时的推荐流程

以后不管是 ETF 还是小盘股，都按这个顺序：

1. 先写假设
- 例如“20 日涨很多再加速，短期回调概率更大”

2. 再写规则配置
- 例如 `ret3 > 0.06 and ret20 > 0.12`

3. 再用券商/PTrade 历史数据跑验证

4. 最后才把规则写进策略

也就是说：
- 先提出假设
- 再验证
- 最后编码
